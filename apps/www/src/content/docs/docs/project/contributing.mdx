# Contributing

Contributions should keep URK small, explicit, and readable.

## Before changing code

- read `docs/BOUNDARY.md`
- read `docs/ARCHITECTURE.md`
- check `docs/DECISIONS.md`
- keep `docs/07_URK/` as archive/reference material

## Good contribution shape

- one clear runtime behavior change
- one focused adapter or controller improvement
- one docs improvement that matches the current repo shape
- one example that teaches a single concept
- tests or checks for the affected workspace

## Avoid

- broad platform rewrites
- hidden public API changes
- turning examples into stable npm APIs
- moving product policy into `@urk/core`
