# 1. What URK Is

URK stands for **Universal Runtime Kernel**.

URK is a web-first runtime and orchestration layer for building interactive applications, motion-heavy pages, visual experiences, lightweight games, and structured real-time UI/scene behavior.

URK should be understood as:
- a runtime kernel,
- an orchestration system,
- an adapter-based interaction layer,
- and a structured alternative to fragile app-specific glue code.

It is especially well suited for:
- interactive websites
- scrollytelling experiences
- Three.js-powered scenes
- advanced UI/runtime coordination
- lightweight app and experience orchestration

---

# 2. Why URK Exists

Modern interactive web experiences often become messy because:
- state is scattered,
- rendering code and app logic are tightly fused,
- motion logic is brittle,
- UI and scene systems drift apart,
- and every project reinvents the same orchestration patterns badly.

URK exists to provide a structured runtime foundation where:
- adapters isolate subsystems,
- controllers orchestrate behavior,
- scenes and states remain manageable,
- and complex interactive experiences can be built without turning into spaghetti.

---

# 3. Core Mission

URK’s mission is to provide a modular runtime kernel for orchestrating interactive web experiences across UI, rendering, motion, input, loading, and related app systems.

---

# 4. Core Identity

URK should be treated as:
- a runtime kernel
- an app/experience orchestration layer
- an adapter-based interaction architecture
- a foundation for advanced web experiences

It is not just a state store.

It is not just a Three.js wrapper.

It is not just a React helper library.

---

# 5. What URK Is Not

URK is not:
- a full general-purpose engine like EnHive
- a monolithic frontend framework replacement
- just a visual scripting system
- just a UI toolkit
- or a random utilities package

Those may connect to it, but they are not its core identity.

---

# 6. Core Product Pillars

## 6.1 Runtime-Oriented
URK should manage real runtime behavior, not just static config/state.

## 6.2 Adapter-Based
Subsystems should be attached through adapters with clear roles.

## 6.3 Controller-Driven
Controllers should orchestrate flows and behavior instead of burying logic inside unstructured components.

## 6.4 Web-First
URK should excel in browser experiences first.

## 6.5 Framework-Friendly, Not Framework-Dependent
URK should work standalone and support wrappers/integrations for React and other frameworks later.

## 6.6 Scene + UI Bridge
URK should help unify scene logic, interaction logic, and UI behavior.

---

# 8. Relationship to EnHive

This distinction matters a lot.

## URK
A lighter, web-first runtime kernel for interactive web experiences and orchestration.

## EnHive
A broader determinism-first, adapter-based engine/runtime platform for games, cinematic workflows, and larger creative/runtime systems.

URK should not be described as “basically EnHive.”
And EnHive should not be reduced to “just URK with extra stuff.”

They are related, but not the same product.
