**Project:** URK  
**Version:** 0.1  
**Status:** Product Strategy Draft  
**Last Updated:** 2026-03-31

---

# 1. Purpose

This document defines URK’s product strategy so it does not become either:
- a vague “future runtime platform,”
- or a random open-source code bucket with no sharp identity.

---

# 2. Strategic Reality

URK has strong potential, but only if it stays clear.

It can easily fail by becoming:
- too abstract
- too close to EnHive
- too dependent on one framework
- too small to feel meaningful
- or too broad to ship

So the strategy must stay grounded.

---

# 3. What Makes URK Real

URK becomes real when it has:
- a credible kernel
- a clean adapter model
- a controller pattern that actually improves app structure
- scene/UI orchestration value
- strong support for interactive web experiences
- enough examples or proofs to show why it matters

Not when it has the most conceptual diagrams.

---

# 4. Recommended Build Phases

## Phase 0 — Canonical Foundation
Lock:
- overview
- architecture
- adapter model
- decisions
- product boundary from EnHive
- terminology

## Phase 1 — Kernel MVP
Build:
- core runtime context
- adapter registration
- controller lifecycle
- state/runtime model
- init/shutdown flow
- event hooks

## Phase 2 — Core Adapter Pack
Build:
- threeAdapter
- pointerAdapter
- loadingAdapter
- uiWidgetsAdapter starter
- storageAdapter or minimal persistence path

## Phase 3 — Experience Proofs
Build:
- scrollytelling demo
- scene + UI sync demo
- loading/transition demo
- interactive object/picking demo
- standalone app-shell example

## Phase 4 — Framework Wrappers
Build:
- react-urk starter
- integration helpers
- documentation/examples

## Phase 5 — Tooling / Expansion
Build:
- dev helpers
- diagnostics
- richer controller packs
- template starters
- more adapters
- deeper product integrations later

---

# 5. Strongest Early Differentiators

## 5.1 Scene + UI Runtime Bridge
This is one of the strongest reasons to care.

## 5.2 Adapter Discipline
Makes the system reusable and robust.

## 5.3 Controller-Oriented Structure
Gives runtime behavior a real architecture.

## 5.4 Standalone-First Web Runtime
Keeps the product broader and cleaner than a framework-bound helper.

---

# 6. Positioning Direction

URK should be positioned as:
- a runtime kernel
- a web interaction orchestration layer
- a structured foundation for advanced interactive browser experiences

Not as:
- a full engine
- a framework replacement
- a random animation helper
- or an academic state machine library

---

# 7. Open Source Direction

URK is a strong candidate for open-source core strategy because:
- it benefits from adoption
- it is infrastructure-oriented
- it can demonstrate EonHive technical credibility
- wrappers, tooling, templates, or premium products can grow around it later

The important part is that the open-source core must still be sharply defined.

---

# 8. Summary

URK wins by being:
- clean
- useful
- modular
- adapter-based
- controller-driven
- and obviously better than ad hoc glue code for interactive web experiences

That is the product strategy that makes sense.
